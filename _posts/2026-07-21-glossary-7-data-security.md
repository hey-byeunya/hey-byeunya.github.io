개발 용어 Q&A 일곱 번째 편. 데이터베이스와 로그인 보안 용어를 정리했습니다.

## Q. 쿼리·인덱스·PK 먼저 정리하면?

**A.**
- **쿼리(Query)** — 데이터베이스에 던지는 질문·요청(SQL 등). 도서관 사서에게 "이런 책 찾아줘" 하는 것. 📚
- **인덱스(Index)** — 찾기 쉽게 해주는 색인·위치 번호. 책 뒤의 찾아보기, 배열 순번(0부터), 검색 속도 향상 등.
- **PK(Primary Key)** — 표에서 각 행을 구분하는 고유 대표번호. 주민등록번호 같은 거예요. 🆔

## Q. 해시(Hash)가 뭐예요?

**A.** 데이터를 정해진 길이의 고유값으로 바꾸는 **단방향** 변환이에요. 데이터의 "지문"이죠.

> 믹서기예요 🥤 — 과일(원본)로 주스(해시값)는 만들어도, 주스를 과일로 되돌릴 순 없어요. 같은 입력은 항상 같은 해시, 입력 길이와 무관하게 결과 길이는 항상 같아요.
{: .prompt-tip }

## Q. 비밀번호를 해시만 해서 저장하면 안전한가요?

**A.** **부족해요.** 해커가 흔한 비번의 해시를 미리 계산한 표(레인보우 테이블)로 대조하거든요. 그래서 두 가지를 더해요:
- 🧂 **소금(Salt)** — 랜덤값을 섞어서, 같은 비번도 다른 해시가 되게 함
- 🐢 **키 스트레칭** — 일부러 느리게 수천 번 반복해서 무차별 대입을 막음

실무에선 bcrypt, Argon2 같은 함수가 이걸 자동으로 처리해요.

> 평문 저장 ❌❌ < 단순 해시 ⚠️ < 해시 + 소금 + 반복 ✅
{: .prompt-warning }

## Q. 해시 테이블, ACID는요?

**A.**
- **해시 테이블** — 해시로 저장 위치를 계산해 데이터를 초고속으로 찾는 자료구조. 호텔 룸키(이름 계산 → 방 번호)처럼요. 🏨
- **ACID** — 데이터베이스가 거래를 안전하게 처리하는 4원칙: 원자성·일관성·독립성·지속성. 꼼꼼한 은행원의 원칙이에요. 🏦

## Q. JWT·세션·토큰·쿠키를 한 번에 정리하면?

**A.** 로그인 상태를 유지하는 개념들이에요.

| 용어 | 정체 | 비유 |
|------|------|------|
| **JWT** | 로그인 증명용 토큰 | 🎫 놀이공원 팔찌 |
| **세션** | 서버가 로그인 기억 | 🗄️ 회원 명부(번호표 소지) |
| **토큰** | 사용자가 정보 보관 | 🎫 정보 담긴 팔찌 |
| **쿠키** | 브라우저의 저장 그릇 | 🍪 주머니 |

핵심: **쿠키는 "저장 그릇", 세션/토큰은 "인증 방식"**이라 차원이 달라요. 주머니(쿠키) 안에 번호표(세션)나 팔찌(토큰)를 넣는 거죠.

## Q. TTL과 RLS는?

**A.**
- **TTL (Time To Live)** — 데이터의 유효기간. 시간이 지나면 자동 만료돼요. 우유 유통기한 🥛. 토큰의 "강제 로그아웃 어려움"을 보완해요.
- **RLS (Row Level Security)** — 데이터베이스의 **행 단위 접근 제어**. 각 행에 "누가 볼 수 있는지" 정책을 걸어요. 개인 자물쇠 달린 사물함 🔒. (Supabase에서 특히 중요!)

> 로그인(JWT)으로 신원 확인 + RLS로 데이터 제한 = 강력한 보안 조합이에요.
{: .prompt-tip }
